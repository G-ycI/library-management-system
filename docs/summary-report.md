# 个人总结报告

**项目名称**：图书管理系统  
**开发周期**：2026年7月10日 - 2026年7月19日  
**技术栈**：Next.js + Flask + SQLite + pypinyin

---

## 一、项目概述

本次居家实训项目，我独立完成了一个基于 Next.js 和 Flask 的全栈图书管理系统开发。项目采用前后端分离架构，前端使用 Next.js 构建响应式用户界面，后端使用 Flask 提供 RESTful API 服务，数据库采用轻量级的 SQLite。整个项目从需求分析、架构设计到编码实现、测试部署，历时约10天完成。

项目实现了图书信息的完整生命周期管理，包括：图书列表浏览、拼音首字母搜索、分类筛选、分页浏览、图书详情查看、新增图书、编辑图书、删除图书、数据统计面板等核心功能。删除和编辑操作需密码验证，确保数据安全。系统内置 33 本图书数据，涵盖 33 种不同分类，并提供 51 种预设分类（每种分类配有独立图标），同时支持自定义分类输入。可借数量不能大于馆藏数量，超限时弹出实时提示。搜索支持拼音首字母匹配（如输入"hz"匹配"活着"），并添加了 300ms 防抖优化。前端共3个独立路由页面，后端提供7个API接口，并提供一键启动脚本。项目已成功部署到 Vercel 平台（https://my-library-system-one.vercel.app），完全满足实训考核要求。

## 二、遇到的最大困难及解决过程

### 困难：前后端联调中的跨域问题和数据格式不一致

在项目开发的第三天，当前端页面和后端API都各自开发完成后，我开始进行前后端联调，这时遇到了两个棘手的问题：

**问题一：跨域请求被浏览器拦截**

前端运行在 `localhost:3000`，后端运行在 `localhost:5000`，由于浏览器的同源策略，前端发起的API请求全部被 CORS 策略拦截，控制台报出 `Access-Control-Allow-Origin` 相关错误。

**解决过程**：
1. 首先我在 Next.js 的配置文件中尝试配置代理转发（rewrites），将 `/api/*` 的请求转发到后端地址，这样前端请求同源地址就不会有跨域问题。
2. 但考虑到生产部署时前后端可能部署在不同域名下，代理不是根本解决方案。
3. 最终我在 Flask 后端引入了 `flask-cors` 扩展，通过 `CORS(app)` 一行代码就解决了跨域问题，同时保留了代理配置作为开发环境的补充方案。

**问题二：前后端数据字段命名不一致**

后端返回的字段是蛇形命名（如 `total_copies`、`available_copies`），而我最开始写前端代码时习惯性用了驼峰命名（`totalCopies`），导致数据无法正确渲染。

**解决过程**：
1. 最初我打算在前端做一层字段转换，但这样增加了不必要的复杂度。
2. 经过思考，我决定保持后端的蛇形命名风格（Python 惯例），前端直接使用相同的字段名。
3. 这个小插曲让我意识到，在项目开始前就应该约定好统一的接口数据规范，避免后期返工。

### 另一个挑战：表单验证的边界情况处理

在开发新增图书表单时，可借数量和馆藏数量之间的逻辑关系让我费了一番心思。最开始只做了简单的非空和最小值校验，但测试时发现用户可以输入"可借数量大于馆藏数量"这种不合理的数据。

**解决思路**：
我在前端表单验证中增加了自定义校验规则，确保可借数量不超过馆藏数量，同时后端也做了同样的校验作为双重保障。后来我进一步优化了体验：当可借数量大于馆藏数量时，输入框上方会弹出红色提示条，2秒后自动消失；当馆藏数量变小时，可借数量会自动适配。这让我深刻体会到"前端验证是为了用户体验，后端验证才是安全底线"这句话的含义。

### 新挑战：敏感操作的密码保护与编辑功能实现

在项目后期，我意识到删除和编辑图书属于不可逆的敏感操作，应该增加一层安全保护。

**解决过程**：
1. 我在后端的 DELETE 和 PUT 接口中添加了密码验证逻辑，密码通过请求体传入，错误时返回 403 状态码。
2. 前端在点击"删除"或"编辑"按钮时，先弹出密码验证模态框，验证通过后才能执行操作。
3. 编辑功能的设计比较复杂：需要先验证密码，验证通过后显示编辑表单，表单预填充当前图书信息，提交时再次带上密码调用 PUT 接口。
4. 编辑成功后我遇到了页面数据不刷新的问题，最初直接用返回数据更新状态但页面没变化，最终改为调用 `fetchBook()` 重新从服务器获取数据解决。

这个过程中我学到了：敏感操作必须有二次确认机制，前后端都要做校验；以及 React 状态更新有时需要从服务器重新获取数据才能保证一致性。

### 新挑战：拼音搜索与分页优化

在项目后期，我希望提升搜索体验，让用户输入拼音首字母就能快速找到图书，比如输入"hz"就能匹配到"活着"。

**解决过程**：
1. 我引入了 Python 的 `pypinyin` 库，将书名和作者转为拼音首字母。
2. 最初我尝试同时支持完整拼音和首字母匹配，但发现完整拼音匹配会导致很多误匹配（如"huo"会匹配到"时间简史"中的"shi"），最终决定只保留首字母匹配，更符合用户直觉。
3. 技术难点在于 SQL 无法直接处理拼音转换，我将搜索逻辑改为在 Python 层面过滤：先获取所有图书，再用 `_match_search()` 函数逐个匹配，最后手动分页。
4. 前端添加了 300ms 搜索防抖，避免用户每输入一个字母就发一次请求，通过 `useEffect` + `setTimeout` 实现，将即时输入值和防抖后的搜索词分离管理。

同时，我还遇到了分页空白页的问题：将每页显示数量从6本改为8本后，前端分页计算没有同步更新，导致出现空白的第6页。修复后将 `Math.ceil(total / 6)` 改为 `Math.ceil(total / 8)`，33本图书按每页8本正好5页，问题解决。

这个过程中我学到了：前后端的数据参数必须保持一致；拼音搜索这类 SQL 无法处理的需求，可以在应用层灵活实现；搜索防抖是提升用户体验和减少服务器压力的有效手段。

### 新挑战：线上部署与平台选择

在项目收尾阶段，我需要将项目部署到线上，提供可访问的演示地址。这是我第一次接触 Serverless 部署，遇到了不少挑战。

**最初方案：Vercel + Render**
最开始我计划前端部署到 Vercel，后端部署到 Render，前后端分离。但是 Render 平台注册和登录一直遇到问题，页面长时间 Loading，账号也无法正常登录，试了好几次都没成功。

**调整方案：全部用 Vercel**
Render 走不通后，我决定换个思路——把后端也搬到 Vercel 上，用 Vercel Serverless Functions 来实现 API 接口。这样前后端都在一个平台，部署和管理都更方便。

**实现过程**：
1. 首先我研究了 Vercel Serverless Functions 的工作方式，了解到可以在 `pages/api/` 目录下写接口
2. 然后将 Flask 后端的 7 个 API 接口（列表、详情、新增、编辑、删除、分类、统计）用 Node.js 重写
3. 因为 Vercel 的 Serverless Functions 无法直接连接本地 SQLite，我改用内存数据存储，内置 33 本初始图书数据
4. 添加了拼音首字母搜索的 Node.js 实现，用 `pinyin-pro` 库（后改用自实现的拼音首字母匹配逻辑）
5. 配置 `vercel.json` 文件，设置构建命令和输出目录
6. 将 Root Directory 设置为 `frontend`，Framework Preset 选择 Next.js
7. 点击 Deploy，等待构建完成

**遇到的坑**：
- 最开始配置了 Turso 云数据库的环境变量，但实际上代码用的是内存数据，导致部署时报错说找不到 secret
- 排查了半天才发现是 `vercel.json` 里引用了不存在的环境变量，删掉后就好了
- 项目名重复的问题：第一次部署时说项目名已存在，换了个名字就好了
- 国内访问 Vercel 不稳定，有时能打开有时打不开，这是网络问题不是代码问题

**部署结果**：
最终项目成功部署到 Vercel，访问地址是 https://my-library-system-one.vercel.app。虽然国内访问可能不稳定，但部署本身是成功的，功能也完整。这次部署经历让我了解了 Serverless 架构的基本概念，也学会了如何将传统的前后端分离项目改造为适合 Serverless 平台部署的形式。

## 三、对 AI 辅助编程的心得体会

这次项目开发过程中，我大量使用了 AI 编程助手作为辅助工具，对此有很深的体会。

### AI 带来的效率提升

1. **快速搭建脚手架**：项目初期，AI 可以在几分钟内生成项目结构、基础配置文件和框架代码，比手动搭建节省了大量时间。
2. **样板代码生成**：像 CRUD 接口、表单组件这类有固定模式的代码，AI 生成的质量很高，直接可用。
3. **调试效率提升**：遇到报错时，把错误信息贴给 AI，往往能快速定位问题原因和解决方案，比自己搜索引擎找答案快很多。
4. **知识补充**：对于不太熟悉的 API 或配置项，AI 可以即时给出示例代码，不用频繁翻文档。

### AI 的局限性

1. **业务逻辑理解有限**：对于复杂的业务规则和特殊的交互逻辑，AI 往往理解不到位，生成的代码需要大量修改。
2. **架构设计能力不足**：AI 擅长写具体的函数和组件，但整体架构设计、技术选型这种宏观层面的决策，还是需要人来把控。
3. **容易"一本正经地胡说八道"**：有时候 AI 会生成看似正确但实际有问题的代码，如果不加审查直接使用，可能会埋下隐患。
4. **上下文有限**：当项目代码量变大后，AI 无法记住所有代码细节，给出的建议可能与现有代码冲突。

### 正确的 AI 使用姿势

经过这次项目，我总结出了几条 AI 辅助编程的经验：

1. **AI 是助手，不是替代者**：AI 可以帮我们写代码，但最终的代码质量和项目成败还是由人负责。
2. **先思考再提问**：在向 AI 提问前，自己先想清楚需求和大致思路，这样才能更好地判断 AI 回答的对错。
3. **审查每一行代码**：AI 生成的代码一定要仔细 review，确认逻辑正确、符合项目规范后再使用。
4. **拆解问题**：把大问题拆成小问题，逐步向 AI 提问，效果比一次性丢一个大需求好很多。
5. **保持学习**：不能因为有 AI 就不学习底层原理，只有自己懂，才能用好 AI。

### 对未来的思考

AI 编程工具的普及是不可逆的趋势，与其抵触，不如拥抱。未来的程序员，核心竞争力可能不再是"能写多少代码"，而是"能否清晰地定义问题、设计合理的架构、把控代码质量"。AI 就像挖掘机，它不能代替设计师设计建筑，但能让建筑工人的效率成倍提升。

## 五、项目收获与反思

### 主要收获

1. **全栈开发能力提升**：通过这次项目，我完整经历了从后端API到前端页面的全流程开发，对前后端分离架构有了更深入的理解。
2. **问题解决能力**：遇到了跨域、数据格式、表单验证、拼音搜索、分页空白页等各种问题，通过查找资料和思考，都一一解决了。
3. **工程规范意识**：对 Git 提交规范、代码组织、文档编写等工程化方面的重要性有了切身的体会。
4. **AI 工具运用**：掌握了如何高效地使用 AI 辅助编程，既提高效率又保证质量。
5. **用户体验优化**：通过添加拼音搜索、搜索防抖、数据统计面板等功能，深刻理解了用户体验优化的重要性。
6. **第三方库集成**：学会了引入 pypinyin 等第三方库扩展系统功能，处理 SQL 无法支持的复杂查询需求。

### 不足之处

1. **测试覆盖不足**：项目主要靠手动测试，没有编写自动化的单元测试和集成测试，这是一个明显的短板。
2. **功能相对简单**：目前只实现了图书的基础管理，缺少用户系统、借阅记录等更复杂的功能。
3. **安全性考虑不够**：虽然删除和编辑有密码保护，但密码明文硬编码，未使用哈希存储；没有实现 JWT 认证、接口限流等生产环境必需的安全措施。
4. **移动端适配**：虽然用了响应式布局，但没有在真实移动设备上充分测试。

### 未来改进方向

如果有更多时间，我计划在以下方面继续完善：

1. 添加用户注册登录系统，实现借阅管理功能

## 六、总结

这次居家实训虽然没有了现场的氛围，但对我来说反而是一次很好的锻炼。没有老师的现场指导，所有问题都要自己想办法解决，这个过程虽然有时会让人抓狂，但解决问题后的成就感也是实实在在的。

通过这个图书管理系统项目，我不仅巩固了 Next.js 和 Flask 的技术知识，更重要的是学会了如何独立完成一个完整的项目——从需求分析到架构设计，从编码实现到测试部署。特别是最后的线上部署环节，让我接触到了 Vercel Serverless Functions 这种新的部署方式，对 Serverless 架构有了初步的认识。AI 辅助编程的实践也让我对未来的软件开发方式有了更多思考。

我知道这个项目还有很多不足之处，但它是我认真思考、一行一行写出来的成果。在这个过程中学到的东西，比最终的项目本身更有价值。从最开始的项目搭建，到中间遇到的各种问题（跨域、数据格式、分页、搜索），再到最后的部署上线，每一步都有新的收获。这不仅是一次技术上的锻炼，更是一次独立解决问题能力的培养。
